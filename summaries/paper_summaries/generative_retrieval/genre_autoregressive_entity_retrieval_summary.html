GENRE: Autoregressive Entity Retrieval

Авторы: Nicola De Cao, Gautier Izacard, Sebastian Riedel, Fabio Petroni.

Аффилиации: Facebook AI Research (Meta).

Публикация: ICLR 2021 (Spotlight).

1. Коротко: о чем статья

GENRE — первая система, которая извлекает сущности (entities) путём их авторегрессионной генерации по имени, слева направо, токен за токеном, обусловленной на контекст. Принципиальный вклад: вместо dense retrieval (encode entity → ANN-поиск) используется seq2seq модель на основе BART/mBART, которая генерирует имя сущности напрямую.

Это первая работа, введшая constrained beam search с trie для generative retrieval — механизм, ставший стандартом для всего направления. Работает на задачах entity disambiguation, entity linking и document retrieval.

2. Контекст

Предшествующие системы entity retrieval имели три принципиальных недостатка:

  • Ограниченное взаимодействие: взаимодействие контекста и сущности захватывается только через dot product, теряя тонкие семантические связи между упоминанием в тексте и кандидатом-сущностью.
  • Memory footprint: необходимость хранить dense representations для всех сущностей в корпусе (миллионы векторов для Wikipedia) создаёт значительную нагрузку на память.
  • Проблема негативов: при обучении требуется subsampling hard negatives, что усложняет тренировочный пайплайн и чувствительно к качеству семплирования.

GENRE устраняет все три проблемы за счёт перехода к авторегрессионной генерации: модель может задействовать весь свой параметрический потенциал для захвата контекстно-сущностных связей, не ограничена одним dot product, и не требует явного негативного семплирования.

3. Ключевые компоненты

3.1. Архитектура: BART / mBART

GENRE строится поверх fine-tuned BART (для английского языка) или mBART (для мультиязычного поиска). Модель работает как seq2seq: принимает текст с упоминанием сущности в контексте и генерирует каноническое имя сущности токен за токеном. Для entity linking контекст включает маркеры [START_ENT] и [END_ENT], выделяющие упоминание.

3.2. Constrained Beam Search с Trie

Ключевое техническое нововведение GENRE — constrained decoding через prefix tree (trie). Все допустимые имена сущностей (из Wikipedia) организованы в trie-структуру. При генерации на каждом шаге разрешены только те токены, которые являются валидным продолжением хотя бы одного существующего имени сущности. Это гарантирует два важных свойства:

  • Модель никогда не сгенерирует несуществующую сущность — полное устранение «галлюцинаций» идентификаторов.
  • Вся вычислительная мощность beam search сосредоточена на различении реальных кандидатов, а не на отсеивании невалидных последовательностей.

Именно этот механизм был напрямую заимствован в DSI (constrained decoding для DocID), SEAL (через FM-Index), TIGER и всех последующих production GR системах.

3.3. Pre-training и Fine-tuning

Обучение проводится в два этапа. Сначала модель pre-train-ится на 9 миллионах пар (контекст, имя сущности) из Wikipedia (BLINK dataset от Facebook), что позволяет модели выучить базовые паттерны entity linking. Затем проводится fine-tuning на конкретную задачу: entity disambiguation (выбор правильного значения неоднозначного упоминания) или end-to-end entity linking (одновременное обнаружение и связывание всех упоминаний в тексте).

3.4. Мультиязычность

mBART-версия GENRE поддерживает entity linking на нескольких языках, генерируя английские имена сущностей из Wikipedia в ответ на запросы на любом из поддерживаемых языков. Это важный шаг к мультиязычному generative retrieval.

4. Эксперименты / Coverage

Дополнительно авторы показывают, что GENRE имеет значительно меньший memory footprint по сравнению с конкурирующими системами, использующими dense representations: нет необходимости хранить векторы для всех сущностей корпуса.

5. Сильные стороны

  • Constrained decoding как стандарт. Введённый trie-based constrained beam search стал архитектурным стандартом для всего направления generative retrieval — это основной технический вклад работы.
  • Богатое контекстно-сущностное взаимодействие. Seq2seq модель может задействовать всю силу Transformer attention для связи контекста и генерируемого имени, в отличие от dot product в dense retrieval.
  • Компактность. Не требует хранить миллионы dense vectors — вся «база знаний» закодирована в параметрах модели плюс компактный trie.
  • Универсальность задач. Единая архитектура покрывает entity disambiguation, entity linking и document retrieval.

6. Ограничения

  • Фиксированное множество сущностей. Trie строится из заранее известных имён сущностей. Добавление новой сущности требует перестройки trie (хотя это дешевле, чем переобучение модели).
  • Скорость инференса. Авторегрессионная генерация по символам медленнее, чем ANN-поиск по dense vectors, особенно для длинных имён сущностей и большого beam size.
  • Зависимость от имён сущностей. Модель опирается на канонические имена из Wikipedia; сущности без устоявшихся имён или с полисемией в именах обрабатываются хуже.
  • Масштаб корпуса. При миллионах уникальных сущностей trie становится очень большим, а constrained decoding — медленнее.

7. Почему важно прочитать

GENRE — вторая ключевая работа после DSI в фундаменте generative retrieval. Именно GENRE вводит constrained decoding через trie как стандартный компонент: этот механизм затем напрямую используется в DSI (trie для DocID), SEAL (FM-Index как обобщение trie), TIGER и всех production GR системах. Также GENRE демонстрирует, что generative retrieval работает не только для document retrieval, но и для entity linking — задачи, структурно более близкой к рекомендациям (сущность = item, контекст = user history). Понимание GENRE существенно облегчает чтение всей последующей литературы по дизайну идентификаторов и constrained decoding.
