Unifying Generative and Dense Retrieval for Sequential Recommendation

← К общему списку статей

Kai Mei, Wenjie Wang, He Zhu, Yuru Li, Tat-Seng Chua (NUS) + Meta AI + JKU Linz — arXiv:2411.18814 — 2024

1. Коротко: о чём статья

Статья задаёт конкретный вопрос: когда generative retrieval (GR) лучше dense retrieval (DR) и когда хуже? — и отвечает на него через систематический экспериментальный анализ на нескольких датасетах. На основе этого анализа авторы предлагают TIGER-Hybrid — fusion-модель, объединяющую оба подхода через late fusion.

Ключевой вывод: GR и DR делают ошибки на разных типах items, их ошибки слабо коррелируют, и поэтому объединение candidate lists улучшает Recall по сравнению с каждым подходом в отдельности. GR особенно силён на cold-start и textually rich items; DR — на популярных items с богатой interaction history.

2. Контекст

После выхода TIGER на NeurIPS 2023 возник закономерный практический вопрос: в каких условиях стоит заменять (или дополнять) сложившийся DR-пайплайн на GR? Оба подхода решают задачу candidate recall — первого этапа воронки рекомендаций, — но делают это принципиально разными способами:

Аспект Dense Retrieval (DR) Generative Retrieval (GR)
Представление item Dense вектор в пространстве размерности d Дискретный SID — последовательность токенов
Механизм поиска ANN (HNSW, FAISS) по cosine similarity Autoregressive beam search с constrained decoding
Сигнал для обучения Contrastive loss по interaction парам Cross-entropy на SID-последовательность
Обновление нового item Нужен embedding (быстро) Нужно переобучение или fine-tuning (медленнее)

DR хорошо работает для популярных items (много обучающих сигналов → точный embedding) и плохо для cold-start (мало взаимодействий → слабый item embedding). GR хорошо работает при богатом текстовом контенте (SID захватывает семантику) и плохо при SID-коллизиях (несколько items с одинаковым или близким SID).

3. Экспериментальный анализ: где выигрывает каждый подход

GR превосходит DR при:

  • Rich textual content: items с детальным описанием, жанрами, тегами — RQ-VAE создаёт информативные SID-ы, beam search находит семантически близкие items, которые не попадают в топ ANN по collaborative вектору.
  • Cold-start / new items: новый item не имеет истории взаимодействий → его DR embedding слабый. SID назначается на основе содержательных признаков (текст, изображение) → GR может находить новый item даже при отсутствии взаимодействий.
  • Хорошее SID-разделение: когда SID хорошо разделяет семантически схожие items (низкий collision rate), beam search точно локализует нужный item в пространстве идентификаторов.

DR превосходит GR при:

  • High-popularity items: популярные items имеют тысячи взаимодействий → их DR embedding обучен на богатом сигнале → ANN точно находит их по user preference вектору. GR при этом может занижать вероятность таких items, если они делят SID-префикс с менее релевантными.
  • SID collision: несколько items, которые получили одинаковый или близкий SID (что неизбежно при конечном словаре RQ-VAE), — beam search выбирает один из них недетерминированно относительно пользовательского контекста.
  • High collision rate: при высоком SID collision rate GR систематически теряет precision — это структурная проблема, не решаемая тюнингом модели.

Конкретные числа (приближённые, Recall@10):

Метод Amazon Beauty Amazon Sports Amazon Books
DR (SASRec two-tower) ~0.140 ~0.118 ~0.092
GR (TIGER) ~0.152 ~0.124 ~0.088
TIGER-Hybrid ~0.171 ~0.139 ~0.097

На Amazon Books (более разреженный датасет, меньше текстового контента) GR уступает DR, но Hybrid всё равно даёт прирост за счёт дополнения coverage. На Beauty и Sports GR лидирует, Hybrid усиливает оба.

4. Анализ failure modes

GR failures

  • SID collision: несколько items получают одинаковый SID (или начало SID совпадает). Beam search генерирует общий SID-префикс, а затем с равной вероятностью выбирает любой из коллидирующих items. Это систематическая ошибка, не связанная с качеством пользовательского представления.
  • Popularity bias в generation: частые SID-ы (соответствующие популярным items или popular SID-префиксам) чаще генерируются — стандартная проблема probability calibration языковых моделей.
  • Out-of-distribution SID: при слабом обучающем сигнале модель может генерировать SID, который был в каталоге при обучении, но давно устарел (удалённые items). Constrained decoding это предотвращает, но лишь при условии актуального trie.

DR failures

  • Cold-start: новые items с малым числом взаимодействий имеют неточный embedding → не попадают в топ ANN даже при семантической релевантности запросу.
  • Semantic mismatch: collaborative filtering сигнал (кто ещё покупал/смотрел) не всегда совпадает с content-based семантикой. Если user preference вектор обучен преимущественно на collaborative сигнале, items с похожим контентом, но разной аудиторией не находятся.
  • ANN approximation error: HNSW и подобные методы дают approximate результаты. На хвосте распределения (long-tail items) приближение работает хуже.

Ключевой вывод об ошибках

Множество items, на которых ошибается GR, и множество items, на которых ошибается DR, имеют низкое пересечение. Это означает, что поздний fusion их candidate lists даёт complementary coverage и улучшает суммарный Recall.

Авторы измеряют overlap между GR и DR candidate lists (доля items, попавших в топ-K обоих методов) и показывают, что он значительно ниже 100%, что и обосновывает выигрыш Hybrid.

5. TIGER-Hybrid: метод объединения

GR-компонент

TIGER-style autoregressive generation SID через beam search. Для каждого user context производится beam search с шириной W, получая топ-K кандидатов с нормализованными generation log-probabilities:

\[ \text{score}_{\text{GR}}(i) = \frac{1}{L} \sum_{t=1}^{L} \log p(s_t \mid s_{<t}, u) \]

DR-компонент

Two-tower dense retrieval с SASRec-like user encoder и item content encoder. ANN поиск (FAISS) по cosine similarity даёт топ-K кандидатов со скорами dot product между user и item векторами.

Late fusion

Candidate lists объединяются через взвешенную комбинацию нормализованных скоров:

\[ \text{score}_{\text{hybrid}}(i) = \alpha \cdot \tilde{s}_{\text{GR}}(i) + (1 - \alpha) \cdot \tilde{s}_{\text{DR}}(i) \]

где \(\tilde{s}\) — нормализованные (min-max или softmax) скоры в пределах своего candidate list, \(\alpha \in [0, 1]\) — параметр смешивания, настраиваемый на validation set. Items, попавшие только в один из списков, получают нулевой скор от другого компонента.

Итоговый список ранжируется по score_hybrid, из него берётся топ-K для передачи на следующий этап воронки (ranking).

6. Почему late fusion лучше early fusion

Интуитивно кажется, что объединение GR и DR ближе к источнику (в пространстве логитов или эмбеддингов) должно давать лучший результат. Авторы объясняют, почему late fusion предпочтительнее:

  • Несовместимые пространства скоров: GR scores — это log-probabilities условных распределений над токенами (sequential, conditional); DR scores — это cosine similarities в embedding space (direct, unconditional). Прямое сложение или конкатенация несмысленны без тщательной калибровки.
  • Специализация компонентов: late fusion позволяет каждому компоненту обучаться независимо на своей objective функции, не компрометируя ни один из них. GR оптимизируется под SID generation, DR — под similarity matching.
  • Простота и интерпретируемость: late fusion легко диагностируется — можно измерить вклад каждого компонента отдельно и понять, какой из них даёт прирост на каком сегменте каталога.

Адаптивное взвешивание

Авторы отмечают возможность адаптивно менять параметр смешивания в зависимости от контекста:

  • Если item новый (мало или нет взаимодействий) — upweight GR компонент.
  • Если item популярный (много взаимодействий) — upweight DR компонент.
  • Если SID collision rate высокий для данного prefix — downweight GR для соответствующих кандидатов.

Простейший способ реализации: lookup table по popularity tier с соответствующим весом, применяемая при fusion scoring.

8. Сильные стороны

  • Чёткая постановка исследовательского вопроса: вместо очередного «наш метод лучше» авторы задают вопрос «когда и почему» — это принципиально более полезно для практиков, выбирающих подход под конкретный каталог.
  • Честный анализ failure modes: обе стороны (GR и DR) разобраны с конкретными механизмами ошибок, без приукрашивания слабостей собственного метода.
  • Практически применимый результат: late fusion прост в реализации и не требует переобучения ни одного из компонентов — можно добавить поверх существующей DR-системы как дополнительный recall path.
  • Диагностический инструментарий: метрики candidate overlap, collision rate, coverage по new/tail items — полезные инструменты для оценки целесообразности Hybrid на собственных данных.

9. Ограничения

  1. Два отдельных компонента: TIGER-Hybrid требует поддержки двух независимых serving paths — GR (LLM + trie) и DR (two-tower + ANN index). Это вдвое большая инфраструктурная сложность по сравнению с single-model подходом.
  2. Настройка параметра смешивания: оптимальное смешивание зависит от датасета и распределения items. Требуется validation data и периодическая перенастройка при изменении каталога или поведенческих паттернов пользователей.
  3. Нет production A/B тестов: все эксперименты проведены на офлайн-метриках (Recall, NDCG) на Amazon Review и MovieLens датасетах. Нет данных о влиянии на CTR, engagement или revenue в реальной системе.
  4. Масштаб датасетов: Amazon Review датасеты — десятки–сотни тысяч items. Industrial каталоги могут быть на 2–3 порядка больше; неясно, сохраняется ли аналогичный прирост от Hybrid при 10M+ items.
  5. Статичный вес смешивания: в основных экспериментах параметр одинаков для всех items и пользователей. Адаптивное взвешивание упомянуто, но детально не оценено количественно.

10. Практические выводы

Когда стоит использовать Hybrid GR+DR

  • Каталог содержит mix cold/warm items — есть постоянный приток новых items наряду с популярными старыми.
  • SID collision rate выше 5–10%: если значительная доля items в каталоге коллидирует по SID, чистый GR теряет precision на этих items, и DR может их компенсировать.
  • DR и GR recall слабо коррелируют: измерить candidate overlap между DR-топ-K и GR-топ-K на holdout set. Если overlap менее 50%, Hybrid почти наверняка даст прирост Recall.
  • Нужен улучшенный хвостовой coverage: tail items систематически хуже представлены в DR из-за малого числа взаимодействий; GR с content SID может частично восстановить их coverage.

Метрики для мониторинга Hybrid системы

Метрика Что показывает Желаемое поведение
GR vs DR candidate overlap Complementarity двух компонентов Низкий overlap (менее 50%) означает высокую ценность Hybrid
SID collision rate в GR path Структурная проблема GR-компонента Мониторить рост при добавлении items в каталог
Coverage новых и tail items Способность находить cold-start items GR coverage new items должен превышать DR
Доля популярных items в топ-K по GR и DR Popularity bias каждого компонента DR чаще находит popular, GR — tail и new
Оптимальный вес по сегментам каталога Балансировка вклада компонентов Разные сегменты могут требовать разного смешивания

Простой путь к внедрению

  1. Запустить TIGER (или аналог GR) параллельно с существующим DR-пайплайном в офлайн-режиме.
  2. Измерить candidate overlap на holdout данных — если менее 50%, Hybrid потенциально ценен.
  3. Реализовать late fusion с фиксированным весом 0.5 как baseline и оценить офлайн по Recall, NDCG.
  4. Подобрать оптимальный вес на validation set через grid search или байесовскую оптимизацию.
  5. Если офлайн прирост значимый — запустить A/B тест в production для оценки онлайн-метрик.
